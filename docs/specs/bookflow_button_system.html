Audit findings

What we found across existing designs.

Six discrete button styles existed across the wireframes, hi-fi designs, and splash screens. Some were canonical, some were ad-hoc inline styles, and we were missing every interactive state. This audit names what's working and what needs to be unified.

The canonical system

Three sizes, four variants.

Every button in Bookflow is a combination of one size and one variant. No exceptions. If a context seems to need a fifth variant, it doesn't — find the closest match and use it.

Sizes

Standard is the default, used in 90% of contexts. Compact is for inline placements within cards (Library hero, list items). Large is reserved for hero CTAs (landing page, paywall primary).

Variants

Four hierarchies of action.

Primary draws the eye to the desired action. Secondary offers an alternative without competing. Tertiary is for low-emphasis paths. Destructive is reserved for irreversible actions only.

Primary · forest filled

Used for the main action on any screen. Maximum one primary button per surface. Filled forest-800 with cream-50 text.

Secondary · outlined

Alternative actions that still warrant visual presence. Used in pairs with primary, never on its own.

Tertiary · text only

Low-emphasis actions. No background, no border. Forest text gives a subtle visual marker without competing with primary actions.

Destructive · for irreversible actions

Reserved for permanent deletions or account-level destructive actions. Red text on transparent background. Always paired with a confirmation dialog before the action fires.

Interactive states

Every button, in every state.

Five states each variant must support: enabled (default), hovered, pressed, disabled, and loading. The loading state replaces the label with a spinner — the only place a spinner is allowed in the system, since the user explicitly initiated the action and is waiting for it.

Icon buttons

Circular, single-purpose.

Used in headers, action bars, and compact list rows. Two sizes (40px standard, 32px small), two variants (default and primary). Icon size is always 18px regardless of button size — keeps tap target large while keeping iconography readable.

Layout patterns

When buttons stack, when they sit side by side.

Three patterns cover 95% of cases. The decision tree: if both actions have similar weight and short labels, use the side-by-side pair. If one action is clearly primary, stack vertically with primary on top. Full-width block buttons are reserved for narrow contexts and final-step CTAs.

Pattern A · Side-by-side pair

Two short, equal-weight actions. Used in confirmation dialogs, modal footers. Gap: 8px. Both buttons grow equally.

Pattern B · Stacked with tertiary

Primary is the main path; secondary action is genuinely secondary. Gap: 6px. Primary is full-width, tertiary sits below at natural width.

Pattern C · Full-width block

Single primary action at the bottom of a screen or modal. Used in onboarding, paywalls, single-step CTAs.

Usage rules

What to never do.

Six rules that hold the system together. Breaking any of them creates the kind of inconsistency this audit was designed to fix.

  1. 01
    Never more than one primary button per surface.
    Two primaries = no primary. The visual hierarchy collapses and the user has to think.
  2. 02
    Never use inline styles to make a one-off button.
    If the system doesn't have what you need, the system needs updating — not the screen.
  3. 03
    Disabled buttons should still be visible, never hidden.
    Visible but disabled tells users "this exists, but you can't do it yet." Hidden teaches them nothing.
  4. 04
    Loading state always uses the spinner — never replace label text with "Loading…"
    The spinner is universal and instant. Text changes feel laggy and translate poorly.
  5. 05
    Button labels are sentence case, verb-led, never truncated.
    "Add to library" not "Add To Library." If a label doesn't fit, the layout is wrong, not the label.
  6. 06
    Destructive actions always require confirmation.
    A destructive button never fires its action immediately — it opens a dialog where the user reconfirms.
